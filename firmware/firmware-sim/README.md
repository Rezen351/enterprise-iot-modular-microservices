# firmware-sim

Python simulator that **reproduces the ESP32 aeroponic-node firmware** (see
`firmware/aeroponic-node/`) over MQTT. Use it to test the full stack —
discovery, pairing, telemetry ingest, actuator control, inputs/outputs — without
real hardware, and to run many **cloned devices with fixed but distinct IDs**.

## What it mimics (MQTT contract, prefix `smartfarm`)

| Topic | Direction | Payload |
|-------|-----------|---------|
| `smartfarm/discovery` | device → broker | `{"node_id","mac","ip","fw_version","status":"online","paired"}` |
| `smartfarm/status/<node_id>` | LWT + retained | `{"status":"offline"/"online","mac",...}` |
| `smartfarm/<node_id>/telemetry` | device → broker | full payload (inputs/outputs/modbus/network/device_info) |
| `smartfarm/actuator/<node_id>` | broker → device | `{"action":"set_output","target","value","req_id"}` |
| `smartfarm/<node_id>/confirm` | device → broker | `{"req_id","target","value","status":"executed"}` |
| `smartfarm/<node_id>/alert` | device → broker | `{"alert":"EMERGENCY_SHUTDOWN","node_id","uptime_s"}` |
| `smartfarm/<node_id>/pair` | broker → device | `{"paired":true}` handshake (persisted) |
| `smartfarm/<node_id>/paired` | device → broker | `{"node_id","paired"}` |
| `smartfarm/<node_id>/sim` | broker → device | test hooks: `emergency`, `clear_emergency`, `set_input`, `set_output`, `discover` |

The telemetry payload shape is identical to `HardwareManager.cpp` so the Module
Service ingests it unchanged. Local-control rules (hysteresis) and the emergency
shutdown are also simulated.

## Install

```bash
cd firmware/firmware-sim
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

> Uses the project broker from `.env`: `MQTT_URL=tcp://mosquitto:1883`.
> Override per instance with `--mqtt-server`.

## Quick start

```bash
# from firmware/firmware-sim/
# 1. clone a device with a FIXED, distinct id  (node-01 auto if none given)
python -m firmware_sim clone --node-id node-07

# 2. run it — it connects, publishes discovery + status, streams telemetry,
#    and listens for actuator commands
python -m firmware_sim run --node-id node-07

# 3. in another terminal, watch it on the broker
mosquitto_sub -h mosquitto -p 1883 -t 'smartfarm/#' -v
```

## Cloning many devices (fixed but different IDs)

Each `clone` writes `instances/<node_id>.json` once, with a randomly generated
MAC and the chosen `node_id`. That identity is **permanent** for that instance,
so every clone is a distinct, stable device:

```bash
# one at a time (explicit ids)
python -m firmware_sim clone --node-id node-08
python -m firmware_sim clone --node-id node-09

# or spawn N at once, all running concurrently in one process
python -m firmware_sim spawn --count 3

# list / inspect / remove
python -m firmware_sim list
python -m firmware_sim info --node-id node-07
python -m firmware_sim delete --node-id node-07
```

All instances share the broker but have unique `node_id` + `mac`, so the Module
Service sees them as separate nodes and you can pair each independently.

## Testing input / output / pairing / discovery

- **Discovery / pairing:** start `run`, then on the Module Service the node
  appears under *discovered*. Pair via the API; the device persists `paired`
  and re-announces it on reconnect.
- **Output (actuator):** publish a command, the sim executes it and ACKs:
  ```bash
  mosquitto_pub -h mosquitto -p 1883 -t smartfarm/actuator/node-07 \
    -m '{"action":"set_output","target":"relay_pump","value":1,"req_id":"abc"}'
  # device replies on smartfarm/node-07/confirm
  ```
- **Emergency stop:** `mosquitto_pub ... -t smartfarm/actuator/node-07 -m '{"action":"emergency_stop","req_id":"x"}'`
  or via sim hook `smartfarm/node-07/sim` `{"action":"emergency"}`.
 - **Force an input (for threshold testing):**
   ```bash
   mosquitto_pub -h mosquitto -p 1883 -t smartfarm/node-07/sim \
     -m '{"action":"set_input","target":"s_atas_temp","value":40}'
   ```
   Values are **engineering units** (e.g. `40` = 40 °C). The local-control rule
   `overheat_protection` will then turn `cooling_fan` ON.

## Dummy sensors (inputs, modbus, telemetry)

There is no real hardware — every reading is generated by a dummy sensor
model (`firmware_sim/sensors.py`) that does a smooth bounded random walk with a
slow sine drift, so values look like a real aeroponic node. The sensor *type* is
inferred from the input/register **name**:

| name contains | unit | typical range |
|---------------|------|---------------|
| `temp`        | °C   | 18–35 |
| `humid`       | %    | 40–95 |
| `press`       | hPa  | 990–1030 |
| `co2`         | ppm  | 400–1200 |
| `lux`         | lx   | 0–20000 |
| `ph`          | —    | 4–9 |
| `tds`/`ec`    | ppm / mS/cm | configurable |
| `flow`        | L/min| 0–6 |
| `volt`/`current`/`power` (modbus) | V / A / W | realistic |
| `level`       | —    | 0/1 (tank full/empty) |

### Telemetry payload (firmware-shaped)

The `telemetry` object is **byte-compatible with the ESP32 firmware**
(`HardwareManager.cpp`): `inputs`/`outputs`/`modbus` are flat scalar values.

```json
{
  "node_id": "node-07",
  "fw_version": "1.0.0-sim",
  "network":  { "ssid": "sim-wifi", "ip_address": "...", "wifi_rssi": -47 },
  "device_info": { "uptime_s": 12, "cpu_freq_mhz": 240, "free_heap_kb": 210, "flash_size_mb": 4 },
  "connection_stats": { "mqtt_connected": true, "uptime_s": 12 },
  "telemetry": {
    "inputs":  { "s_atas_temp": 1823, "s_bawah_humid": 2415, "water_level": 1, "btn_emergency": 0 },
    "outputs": { "relay_pump": 1, "cooling_fan": 180, "grow_light": 0, "mist_pump": 0 },
    "modbus":  { "power_meter": { "voltage": 221.4, "current": 1.3, "power": 271.5 } }
  },
  "sensors": {
    "s_atas_temp":  { "value": 25.6, "unit": "C" },
    "s_bawah_humid":{ "value": 72.4, "unit": "%" },
    "water_level":  { "value": 1.0,  "unit": "" },
    "btn_emergency":{ "value": 0.0,  "unit": "" }
  }
}
```

`telemetry.inputs` carries the raw ADC value (what the firmware would `analogRead`/
`digitalRead`), while the extra top-level `sensors` field is a human-friendly
extension (engineering value + unit) kept **outside** `telemetry` so the firmware
contract stays intact. `telemetry.modbus` is populated from the default sample
RS485 `power_meter` device (edit `modbus` in the instance JSON to add more).

## Injecting commands (CLI)

Instead of hand-writing `mosquitto_pub`, use the `inject` command to drive a
running device:

```bash
# turn an output ON/OFF (publishes smartfarm/actuator/<id> set_output)
python -m firmware_sim inject --node-id node-07 --target relay_pump --value 1

# PWM level 0..255
python -m firmware_sim inject --node-id node-07 --target cooling_fan --value 180

# force a sensor to a specific engineering value (e.g. 40 °C)
python -m firmware_sim inject --node-id node-07 --target s_atas_temp --value 40 --kind input

# trigger emergency stop (zeros all outputs + publishes alert)
python -m firmware_sim inject --node-id node-07 --kind emergency
```

`--kind` is `output` (default), `input`, or `emergency`.

## Notes

- Instance configs live in `firmware/firmware-sim/instances/` (git-ignored from
  the main repo by default — keep them local). Set `FIRMWARE_SIM_HOME` to
  relocate.
- `paho-mqtt` is the only dependency. No real GPIO is touched.
